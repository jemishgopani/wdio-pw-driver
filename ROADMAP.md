# wdio-pw-driver — production-readiness roadmap

Current status: `0.1.0-alpha.0`. This document tracks everything between here
and a production-grade `1.0.0`. Items are grouped by tier; within each tier
they're ordered by impact.

> "Production-grade" labels are mostly aspirational. The real gate is **real
> users running real suites and not hitting blockers**. The tiers below are
> the table-stakes to get those users — not a guarantee of greatness.

---

## Tier 1 — blockers for beta → production (must)

| # | Item | Effort | Why it matters |
|---|---|---|---|
| 1 | **Flake-free CI for 50+ consecutive runs** | medium | The implicit-timeout flake at `phase2.test.ts:101` on macOS-Node-20 needs root-cause fix — bump the implicit-timeout floor (currently 100 ms) or `waitFor` DOM-ready in `beforeEach`. Any flake erodes trust faster than missing features. |
| 2 | **Publish to npm under the `@beta` tag** | small | `npm publish --tag beta`. Wire into a release workflow. Currently anyone who wants the driver has to clone the repo — direct adoption blocker. |
| 3 | **Automated release pipeline** | small-medium | `changesets` or `release-please` for automated CHANGELOG entries + version bumps + GitHub releases. One PR per release; reproducible publish. |
| 4 | **Documented semver / support policy** | small | Two paragraphs in the README: "what counts as a breaking change", "support policy for major versions". Sets expectations so users can plan upgrades. |
| 5 | **Security scanning enabled** | small | Turn on Dependabot + CodeQL in repo settings. Both free for public repos. |
| 6 | **`SECURITY.md` + `CONTRIBUTING.md`** | small | One page each. SECURITY.md needs an email address for responsible disclosure. CONTRIBUTING.md describes the local dev loop + how to run the suite. |

## Tier 2 — confidence multipliers (should)

| # | Item | Effort | Why it matters |
|---|---|---|---|
| 7 | **Real cross-browser CI (not just smokes)** | medium | Firefox + WebKit currently run a 2-test smoke; expand to the full integration suite per engine. This surfaces real engine-specific bugs that Chromium hides. |
| 8 | **Coverage thresholds enforced** | small | `vitest --coverage` with `--coverage.lines 85`. Fail the build below threshold. Catches dead branches users will eventually exercise. |
| 9 | **Performance benchmarks tracked over time** | medium | `website/docs/benchmarks.md` is the start. Wire the runner into a nightly job, compare each run against the previous baseline, post a delta in CI. Regress detection > vanity numbers. |
| 10 | **Two real external test suites validated** | medium | Beyond `wdio-sample`, pick another public WDIO suite (OrangeHRM, demowebshop) and run it nightly against the driver. One real-world bug surfaced this way is worth ten unit tests. |
| 11 | **Memory leak audit** | medium | Loop 100 `newSession → click → deleteSession` cycles, watch RSS. The `context.close()` deadlock workaround leaks 1 `BrowserContext` per `pwNewContext` — quantify, accept-and-document, or fix upstream. |
| 12 | **Multiremote properly tested** | medium | Currently mostly assumed-to-work. Add an integration test that opens Chromium + Firefox in one WDIO run via multiremote. |
| 13 | **Autogenerated API reference** | medium | TypeDoc pointed at `src/`, published as part of the Docusaurus site. Manual docs go stale; types don't. |

## Stretch — beyond 1.0

These are items that would mark the driver as best-in-class, but aren't gating
production use. Park here so they don't get lost.

- **Network throttling** (`pwThrottleNetwork`) — needs `playwright-core` version bump; `context.throttleNetwork` isn't in our pinned 1.40+.
- **WebAuthn protocol commands** — Playwright has no public API; would require CDP bridge work.
- **Raw CDP passthrough** (`browser.pwSendCDP(method, params)`) — would expose internal `_connection`; large API stability risk to take on.
- **Expect-webdriverio mock matchers** (`toBeRequested`, `toBeRequestedWith`) — needs a `pwRoute`-backed call ledger that exposes WDIO's expected `{calls: [...]}` shape.
- **DevTools-protocol Lighthouse integration** — `wdioPW lighthouse <url>` reusing the launched browser.
- **Auto-update banner** — print a one-line notice in the launcher when a newer driver version is on npm.
- **Trace viewer embed** — serve traces from a built-in HTTP endpoint instead of requiring `npx wdioPW trace <file>`.

## Suggested execution order

**Week 1** — Tier 1 in full. Especially #1 (kill the flake) and #2 (publish to
npm `@beta`). Without those, nothing else moves the needle.

**Weeks 2–4** — Tier 2 in this order: #10 (real-suite validation), #9 (perf
tracking), #7 (real cross-browser). These are the items that surface real bugs.
Coverage thresholds (#8) and TypeDoc (#13) can run in parallel.

**Whenever** — Tier 3 polish, interleaved with user-reported issues.

---

## The real beta → 1.0 signal

Roughly **5–10 real external users running the driver in production-ish settings
for 30 days without filing P0 issues**. That's the actual gate; everything above
is table-stakes to get those users.

When that happens:

1. Bump version to `1.0.0-rc.N` on npm.
2. Document a one-page upgrade guide.
3. Freeze the API surface for `1.0.0`.
4. Tag the release.
